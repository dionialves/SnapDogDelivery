## Visão Geral

O SnapDog Delivery é uma aplicação web para gerenciamento de pedidos de delivery de hot dogs. O sistema é dividido em dois contextos independentes que coexistem na mesma aplicação:

- **Admin**: painel interno para operadores gerenciarem clientes, produtos, pedidos e usuários do sistema.
- **Storefront**: área pública onde clientes podem criar contas, navegar no catálogo, montar um carrinho e finalizar pedidos online.

A versão 0.3.0 marca a **implementação completa do storefront**, tornando o sistema funcional de ponta a ponta: um cliente pode se cadastrar, navegar, comprar e acompanhar seus pedidos sem nenhuma intervenção manual do admin.

## Contexto Admin

Acessível em `/admin/**`. Requer autenticação de usuário administrativo.

### Clientes

Um **Cliente** representa uma pessoa física que realiza pedidos. O admin gerencia clientes diretamente pelo painel.

**Dados obrigatórios:** nome, telefone, e-mail, endereço completo (logradouro, número, bairro, cidade, estado, CEP).

**Regras de negócio:**

- Um cliente não pode ser excluído se possuir pedidos associados a ele. A tentativa resulta em erro de negócio.
- A busca de clientes funciona por nome ou telefone, sem distinção de maiúsculas/minúsculas.
- Todo cliente criado pelo admin começa com o campo `active = true` por padrão.
- A partir desta versão, clientes também possuem credenciais de acesso ao storefront (e-mail + senha). Clientes criados pelo admin não recebem senha automaticamente — eles precisam se cadastrar pelo storefront para obter acesso à área pública.

---

### Produtos

Um **Produto** representa um item do cardápio disponível para venda.

**Dados obrigatórios:** nome, preço.
**Dados opcionais:** descrição, imagem (upload de arquivo), status ativo/inativo.

**Regras de negócio:**

- Somente produtos com `active = true` são exibidos no catálogo público. Produtos inativos ficam visíveis apenas no painel admin.
- O preço é sempre armazenado como `BigDecimal` para garantir precisão monetária.
- A landing page exibe até **6 produtos ativos**, ordenados por nome, como seção de destaques.
- O catálogo público exibe **12 produtos por página**, ordenados por nome.
- Um produto inativo não pode ser adicionado ao carrinho — a tentativa resulta em erro de negócio.
- Produtos podem ser excluídos pelo admin sem restrição de pedidos associados (o preço já foi capturado como snapshot no momento do pedido).

---

### Pedidos

Um **Pedido** registra a compra de um ou mais produtos por um cliente. É a entidade central do sistema.

**Regras de negócio:**

**Criação:**
- Um pedido deve ter ao menos um produto e um cliente válido.
- O preço de cada produto é capturado no momento da criação do pedido (`priceAtTime`). Alterações posteriores no preço do produto não afetam pedidos já realizados.
- Todo pedido nasce com status `PENDING`.
- Pedidos criados pelo admin têm `origin = MANUAL`. Pedidos criados pelo cliente via storefront têm `origin = ONLINE`.
- Pedidos `ONLINE` armazenam um **snapshot do endereço de entrega** no momento da compra. Isso garante que alterações futuras no perfil do cliente não modifiquem o endereço de um pedido já feito.

**Modificação:**
- O conteúdo de um pedido (itens, cliente) **não pode ser alterado** após a criação. Somente o status pode ser atualizado.

**Progressão de status:**
- O status só avança na sequência definida. Não é permitido pular etapas.
- Cancelamento só é permitido enquanto o pedido está em `PENDING`.
- Pedidos nos status `DELIVERED` ou `CANCELED` são **finais** e não aceitam mais nenhuma alteração.

**Exclusão:**
- O admin pode excluir um pedido sem restrições de status (operação administrativa).

---

### Usuários Administrativos

Um **Usuário Administrativo** é uma conta de acesso ao painel `/admin`. Não deve ser confundido com um Cliente.

**Regras de negócio:**

- O e-mail é único no sistema. Não é possível cadastrar dois usuários com o mesmo e-mail.
- Senhas são armazenadas com hash BCrypt. Nunca em texto plano.
- Um usuário **não pode excluir a própria conta** enquanto estiver autenticado.
- Ao atualizar um usuário, a senha só é alterada se um novo valor for fornecido. Se o campo de senha for deixado em branco, a senha atual é mantida.
- Existem três papéis (roles):
  - `ADMIN`: acesso completo ao painel, exceto gestão de usuários.
  - `SUPER_ADMIN`: acesso completo ao painel, incluindo criação, edição e exclusão de outros usuários.
  - `USER`: papel previsto, mas sem permissões configuradas nesta versão (reservado para versões futuras).

---

### Dashboard

O painel inicial do admin exibe métricas operacionais do dia, comparadas com o dia anterior.

**Dados exibidos:**
- Pedidos realizados hoje vs. ontem.
- Receita total gerada hoje vs. ontem.
- Novos clientes cadastrados hoje vs. ontem.
- Ticket médio dos pedidos de hoje vs. ontem.
- Os 5 pedidos mais recentes de toda a base.
- Os 5 produtos mais vendidos (por quantidade total de itens em pedidos).

**Regras de negócio:**
- O ticket médio é calculado dividindo a receita total pelo número de pedidos do dia. Se não houver pedidos, o ticket médio é zero.
- As métricas são sempre relativas ao dia corrente (sem filtro por status de pedido).

## Contexto Storefront (Área Pública)

Acessível em `/`, `/catalog`, `/cart`, `/checkout` e `/account`. Parte das rotas exige autenticação de cliente.

### Catálogo

A área pública começa pelo catálogo, acessível sem login.

**Regras de negócio:**
- A landing page (`/`) exibe até 6 produtos em destaque (ativos, ordenados por nome).
- O catálogo completo (`/catalog`) lista todos os produtos ativos, paginados (12 por página), em ordem alfabética.
- A página de detalhe de um produto (`/catalog/{id}`) só é acessível se o produto estiver ativo. Tentar acessar um produto inativo retorna erro 404.

---

### Carrinho

O carrinho é mantido **na sessão HTTP** do navegador. Não é persistido no banco de dados.

**Regras de negócio:**
- O carrinho existe enquanto durar a sessão do navegador. Ao fechar o navegador ou a sessão expirar, o carrinho é perdido.
- Apenas clientes autenticados podem adicionar itens ao carrinho e acessar a página do carrinho.
- Não é possível adicionar um produto inativo ao carrinho.
- Se o mesmo produto for adicionado mais de uma vez, a quantidade é somada ao item já existente.
- Atualizar a quantidade de um item para zero ou negativo remove o item do carrinho.
- O número de itens no carrinho é exibido no cabeçalho de todas as páginas da área pública.

---

### Checkout

O checkout converte o carrinho em um pedido persistido no banco de dados.

**Regras de negócio:**
- Apenas clientes autenticados podem acessar o checkout.
- Não é possível finalizar um pedido com carrinho vazio.
- O preço dos itens é capturado no momento da confirmação do pedido, a partir do preço atual de cada produto no banco de dados. O preço exibido no carrinho é apenas informativo.
- O endereço de entrega é extraído do cadastro do cliente no momento da confirmação e salvo como snapshot no pedido.
- Após a criação bem-sucedida do pedido, o carrinho é esvaziado automaticamente.
- O pedido criado via checkout sempre tem `origin = ONLINE`.

---

### Área da Conta

Acessível em `/account/**`. Requer autenticação de cliente.

**Regras de negócio:**
- O painel da conta exibe os **5 pedidos mais recentes** do cliente autenticado.
- A listagem completa de pedidos é paginada (10 por página), em ordem decrescente de data.
- Um cliente só pode visualizar os próprios pedidos. Tentar acessar um pedido de outro cliente retorna erro 404.
- O cliente pode editar seus dados cadastrais (nome, telefone, endereço), mas **não pode alterar a senha** pela área da conta nesta versão.

---

### Autenticação do Cliente

**Regras de negócio:**
- O cadastro exige e-mail único na base de clientes. Não é possível criar duas contas com o mesmo e-mail.
- A senha é armazenada com hash BCrypt.
- O login é feito via e-mail e senha. Após autenticação bem-sucedida, o cliente é redirecionado ao catálogo.
- O sistema de autenticação de clientes é **completamente independente** do sistema de autenticação de usuários admin. As duas bases de dados de credenciais são separadas e não se misturam.

## Fluxos de Trabalho

### Jornada do Cliente: do catálogo ao pedido confirmado

```
1. Acessa a landing page (/)
      └── Vê produtos em destaque

2. Navega pelo catálogo (/catalog)
      └── Filtra e encontra o produto desejado

3. Acessa o detalhe do produto (/catalog/{id})
      └── Clica em "Adicionar ao carrinho"
            ↓
      [Se não estiver logado]
            └── Redirecionado para /login
                  └── Tem conta? → Faz login
                  └── Não tem conta? → Acessa /register → Preenche cadastro → Redirecionado para /login

4. Com sessão autenticada, o item é adicionado ao carrinho
      └── Contador no cabeçalho é atualizado

5. Acessa o carrinho (/cart)
      └── Revisa itens, quantidades e total
      └── Pode ajustar quantidades ou remover itens

6. Clica em "Finalizar Pedido" → Revisão de checkout (/checkout)
      └── Vê resumo do pedido e endereço de entrega cadastrado

7. Confirma o pedido (/checkout/confirm)
      └── Pedido criado com status PENDING e origin ONLINE
      └── Carrinho esvaziado
      └── Redirecionado para tela de confirmação (/checkout/confirmation/{id})

8. Acompanha o pedido em /account/orders
      └── Pode clicar no pedido para ver o detalhe com status atual
```

---

### Jornada do Admin: do login ao pedido entregue

```
1. Acessa /admin/login
      └── Autentica com e-mail e senha de usuário admin

2. Redirecionado ao Dashboard (/admin/dashboard)
      └── Vê métricas do dia: pedidos, receita, novos clientes, ticket médio
      └── Vê os 5 pedidos mais recentes e os 5 produtos mais vendidos

3. Acessa a lista de pedidos (/admin/orders)
      └── Pode filtrar por status e por nome/telefone do cliente
      └── Vê pedidos mais recentes primeiro

4. Abre o detalhe de um pedido em PENDING
      └── Revisa os itens e o cliente

5. Avança o status para PREPARING
      └── Confirma que a preparação começou

6. Avança o status para OUT_FOR_DELIVERY
      └── Confirma que o pedido saiu para entrega

7. Avança o status para DELIVERED
      └── Pedido marcado como entregue — status final, não pode mais ser alterado
```

---

### Fluxo de Status de Pedido

```
                  ┌──────────┐
    Criação ────► │ PENDING  │
                  └────┬─────┘
                       │ avança          cancelar (somente daqui)
                       ▼                 ────────────────────────►  CANCELED (final)
                  ┌──────────┐
                  │PREPARING │
                  └────┬─────┘
                       │ avança
                       ▼
              ┌─────────────────┐
              │ OUT_FOR_DELIVERY│
              └────────┬────────┘
                       │ avança
                       ▼
                  ┌──────────┐
                  │DELIVERED │  ◄── status final
                  └──────────┘
```

**Regras resumidas:**
- Cada status só avança para o próximo na sequência. Não é possível pular etapas.
- Cancelamento só é permitido a partir de `PENDING`.
- `DELIVERED` e `CANCELED` são estados terminais: nenhuma alteração é aceita a partir deles.

## Segurança e Controle de Acesso

O sistema opera com **duas cadeias de segurança independentes**, configuradas no Spring Security.

### Cadeia Admin (Ordem 1)

| Rota | Acesso permitido |
|---|---|
| `/admin/login` | Público |
| `/admin/users/**` | Somente `SUPER_ADMIN` |
| `/admin/**` (demais) | `ADMIN` ou `SUPER_ADMIN` |

- Credenciais lidas da tabela `tb_users`.
- Login via e-mail e senha.
- Após login bem-sucedido: redireciona para `/admin/dashboard`.

### Cadeia Pública (Ordem 2)

| Rota | Acesso permitido |
|---|---|
| `/`, `/catalog/**` | Público (sem login) |
| `/login`, `/register` | Público (sem login) |
| `/cart/**` | Somente `CUSTOMER` autenticado |
| `/checkout/**` | Somente `CUSTOMER` autenticado |
| `/account/**` | Somente `CUSTOMER` autenticado |

- Credenciais lidas da tabela `tb_customers`.
- Login via e-mail e senha.
- Após login bem-sucedido: redireciona para `/catalog`.

### Isolamento entre as cadeias

As duas cadeias não compartilham contexto de autenticação. Um usuário admin autenticado em `/admin` não possui acesso à área de cliente (`/account`) e vice-versa. As senhas de ambos os grupos são codificadas com BCrypt.

## Glossário

| Termo | Definição |
|---|---|
| **Customer** | Cliente que realiza pedidos. Possui cadastro com endereço e credenciais de acesso ao storefront. |
| **User** | Usuário administrativo com acesso ao painel `/admin`. Não é um cliente. |
| **Order** | Pedido de compra. Associa um cliente a um ou mais produtos com preço capturado no momento da venda. |
| **Product** | Item do cardápio. Pode estar ativo (visível no catálogo) ou inativo (oculto do público). |
| **Cart** | Carrinho de compras temporário, mantido na sessão HTTP. Não é persistido no banco de dados. |
| **Checkout** | Processo de conversão do carrinho em um pedido persistido. |
| **OrderStatus** | Situação atual de um pedido: `PENDING`, `PREPARING`, `OUT_FOR_DELIVERY`, `DELIVERED` ou `CANCELED`. |
| **OrderOrigin** | Indica se o pedido foi criado pelo admin (`MANUAL`) ou pelo cliente via storefront (`ONLINE`). |
| **priceAtTime** | Snapshot do preço unitário de um produto no momento em que o pedido foi criado. Imutável. |
| **deliveryAddress** | Snapshot do endereço de entrega do cliente no momento do checkout. Presente apenas em pedidos `ONLINE`. |
| **Role** | Papel de um usuário admin: `USER`, `ADMIN` ou `SUPER_ADMIN`. Define o nível de acesso ao painel. |
| **Storefront** | Área pública da aplicação, voltada ao cliente final. |
| **Admin** | Área restrita da aplicação, voltada à operação interna do negócio. |
| **Dual chain** | Arquitetura de segurança com duas cadeias independentes — uma para admin, outra para clientes. |
